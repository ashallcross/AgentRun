## Decision 1: Rename — Architect Question Answers

### Q1: App_Data folder — migration concern?

**No migration concern.** The `App_Data/Shallai.UmbracoAgentRunner/` path is only referenced as defaults in `AgentRunnerOptions.cs` (lines 9 and 19). No hardcoded references elsewhere. V1 hasn't shipped — no production instances exist. TestSite data is dev artefacts.

**Action for story:** Delete old `App_Data/Shallai.UmbracoAgentRunner/` folder after verifying rename works. No migration logic needed.

### Q2: Composer class rename?

**Yes — rename to `AgentRunComposer`.** It's the DI entry point, visible in stack traces and Umbraco's composition logging. A class called `AgentRunnerComposer` in a package called `AgentRun.Umbraco` looks like a leftover.

### Q3: AgentRunnerOptions rename?

**Yes — rename to `AgentRunOptions`.** Drop "Runner" from all classes that carried the old compound name:

- `AgentRunnerOptions` → `AgentRunOptions`
- `AgentRunnerComposer` → `AgentRunComposer`
- `AgentRunnerException` → `AgentRunException`

"Runner" was part of "AgentRunner" as a single compound word. The product is now "AgentRun" — keeping "Runner" creates a ghost of the old name.

### Q4: Cross-project .csproj references beyond ProjectReference?

**Yes — `InternalsVisibleTo`** in the main .csproj (line 26): `Include="Shallai.UmbracoAgentRunner.Tests"`. Must update to `AgentRun.Umbraco.Tests`. This is captured in the missing items table above (M7).

### Q5: Does rename affect project-context.md?

**Yes.** It contains namespace patterns, folder conventions, path defaults, element prefixes, API routes, config keys, and bundle IDs — all of which change. **Regenerate it after the rename using the generation skill.** Do not hand-edit. Captured in M13 above.

### Q6: Section permission — migration or documentation?

**Documentation only.** V1 hasn't shipped — no production Umbraco instances with the old section alias. TestSite needs permission re-granted manually after the alias changes.

**Manual E2E step for story:** "Verify the section appears after granting permission to Administrators user group with the new alias `AgentRun.Umbraco.Section`."

## Decision 2: Epic Sequence — Architect Assessment

### Verdict: Approved

The proposed sequence (rename → Epic 9 → 10 → 11 → 12) is architecturally sound.

### Per-Epic Notes

**Pre-Epic 9 rename** — Non-negotiable before docs and packaging. One-time cost, zero logic changes. Correct.

**Epic 9 (docs + packaging)** — No changes. Ships V1 with the right name. Note: Story 9.3 currently references `dotnet add package Shallai.UmbracoAgentRunner` in its acceptance criteria — the SM should note this needs updating to `AgentRun.Umbraco` when the story is created.

**Epic 10 (stability + launch)** — Good prioritisation. Three real bugs:
- Instance concurrency locking (SemaphoreSlim per-instance) — known deferred work
- Context management for long conversations — token waste on multi-step workflows
- Multi-turn interactive fix (ToolLoop exits when LLM asks a question)

Note: Open source licence decision has zero code dependency — it's a `LICENSE` file and `PackageProjectUrl` in the .csproj. Keep it in the epic for tracking but flag it as a product/legal decision.

**Epic 11 (adoption)** — Workflow templates are high-value, low-risk. Tree navigation is a significant UX refactor (V2 doc scope assessment is accurate). Token usage logging as structured log events (not a dashboard) is the right scoping — Pro prerequisite without being a Pro feature.

**Epic 12 (Pro foundation)** — Architecturally correct. The `IInstanceStorageProvider` abstraction was considered for earlier placement (it's a pure refactor), but there's no benefit until the database provider ships. Keeping it in Epic 12 avoids adding abstraction risk to the stability epic.

The V2 future considerations doc's interface design (sections 1-2) still aligns with V1's implementation. The `IInstanceManager` and `IConversationStore` interfaces are the natural seam points — the storage provider wraps the I/O beneath them.

### One Flag for Epic 12

The package split (`AgentRun.Umbraco` + `AgentRun.Umbraco.Pro`) is straightforward if the Engine boundary stays clean — and it has. But the story spec for the split should explicitly call out:
- The Pro project references the free project (never the other way around)
- The free project's public API surface must be reviewed before the split to ensure stability
- Breaking changes after the split affect two packages

### V2 Doc Maintenance

The V2 future considerations doc (`v2-future-considerations.md`) is marked as a living document and still references `Shallai.UmbracoAgentRunner` throughout (package names in commercial model section, MCP package naming). After the rename ships, update the package names in section 6 (commercial model) at minimum — those are forward-looking decisions referenced during Epic 12 planning.
