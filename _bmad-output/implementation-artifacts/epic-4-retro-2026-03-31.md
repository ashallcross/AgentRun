## Challenges & Growth Areas

### 1. Story specs missed failure modes (root cause of 5/8 review catches)
Story specs described happy paths but not failure scenarios. The dev agent implements exactly what's specified — unspecified failure paths become bugs caught only during review.

**Examples:**
- ToolLoop had no iteration cap — spec said "loop until no tool calls" but didn't mention runaway LLM protection
- OCE on cancellation — spec said "catch all exceptions" but didn't distinguish cancellation from errors
- FindIndex returning -1 — spec didn't mention step-not-found scenario

**Root cause:** Story creation process, not implementation quality.

**Action taken:** Added "Failure & Edge Cases" section requirement to project-context.md and memory. Applied to all future story specs.

### 2. Planning assumptions about external APIs
- Story 4.2 spec assumed `IAIChatService.GetChatClientAsync(string profileId)` — actual API is `CreateChatClientAsync(Action<AIChatBuilder>)` with builder pattern
- Fourth instance of planning assumptions being wrong about API shapes (pattern from Epics 1-3)
- Process handles it cleanly — no rework — but could be reduced with pre-implementation API verification

### 3. Deferred work growing (32 items total)
Triaged during this retro:
- **1 fix addresses 5 items** — per-instance SemaphoreSlim solves entire concurrency cluster
- **4 items resolved by Epic 5** — path sandboxing and workflow validation
- **2 items are planned stories** — retry (7.2), section tree UX (Epic 9)
- **1 permanent constraint** — Lit component rendering in tests
- **~18 low-severity edge cases** — no action needed

**Decision:** Defer concurrency locking to post-Epic 9 hardening pass. Single-user testing means zero practical risk.

## Key Insights

1. **The spec is the lever, not the review** — dev agent builds exactly what's specified. Adding failure modes to specs prevents 60%+ of critical review catches.
2. **Error handling edge cases are the consistent blind spot** — happy path always solid, failure/cancellation/boundary paths missed. Now addressed via story spec requirement.
3. **Deferred work looks scarier than it is** — when triaged, most items are either naturally resolved by upcoming work or genuinely low risk.
4. **Live provider testing catches what unit tests can't** — API shape mismatches, middleware behaviour, real-world response patterns.
