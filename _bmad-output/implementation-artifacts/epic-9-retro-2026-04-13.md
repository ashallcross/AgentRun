## What Went Well

1. **Shipped a real product** — Installable NuGet package with native Umbraco backoffice integration. Any developer can `dotnet add package` and get working AI workflows against their own content. Not a demo or proof of concept.

2. **In-process content tools (9.12) unlocked core product value** — Reading directly from the published content cache via `IUmbracoContextFactory.EnsureUmbracoContext()`. Handles 11 property editor types with per-property error isolation. This is what makes it feel like a native Umbraco package.

3. **Test suite grew 59%** — 328 to 521 backend tests. Canary workflow tests (9.2, 9.10) validate that shipped workflows pass the schema and validator. Comprehensive regression coverage.

4. **Code review discipline held** — ~30 patches caught across Epic 9, consistent with Epic 8's finding that reviews are "essential and load-bearing." Zero bugs escaped to production.

5. **Resolution chain pattern (9.6) established and reused** — Mechanism-vs-policy separation for tool limits. Clean architectural foundation reused verbatim in 9.9 for read_file size guard.

6. **Diagnostic-before-fix discipline** — The 5-line FinishReason instrumentation in 9.1b found the real root cause (MaxOutputTokens=4096 silent cap) in minutes, preventing hours of wasted rhythm-restructuring work.

7. **Three-round handoff protocol** — Dev (Amelia) → Docs (Paige) → Human E2E (Adam) proven across 9.3 and other stories. Clean separation of concerns.

8. **Three polished workflows shipping** — Content Quality Audit, Accessibility Quick-Scan, Content Audit (Umbraco-native). Documentation leads with real product framing.

## What Didn't Go Well

1. **fetch_url diversion** — Four stories (9.0, 9.6, 9.7, 9.9) spent hardening external URL crawling before building the actual core feature: in-process content tools. This is an Umbraco package — the headline feature should have been "audit the content in your CMS" from day one, not "fetch external URLs and hope the LLM can parse raw HTML."

2. **Context bloat rediscovered from scratch** — The original app had already solved this problem with the BMAD-style pattern (write intermediate results to disk, read selectively). Documentation and links were available. The team walked into the same wall — 1.8 MB of raw HTML in conversation context causing Sonnet 4.6 empty turns — and spent days diagnosing what was already a known, solved problem.

3. **Mid-epic scope reshaping** — Two SCPs required when live testing invalidated assumptions. The 9.1b reshape from "iterate prompts until trustworthy" to "land AngleSharp server-side extraction first" cost significant time. The original story breakdown assumed the tools would just work.

4. **LLM non-determinism complicated prompt work** — Sonnet 4.6 stalls, empty turns, and non-deterministic HTML parsing made the Content Quality Audit polish loop non-convergent until the architectural fix (structured extraction) landed.

5. **Umbraco API edge cases** — Property editor behaviours (BlockList nesting, CompositionPropertyTypes for inherited properties, IHtmlEncodedString from RTE, templateAlias resolution via IFileService) were discovered through trial-and-error rather than research. Community resources had answers.

---

## Key Insights

1. **The product is for editors and content managers, not developers** — Every priority decision must flow from this. An editor at a client organisation needs workflows that audit their content, not a developer tool for fetching URLs.

2. **Context efficiency is non-negotiable** — Write to disk, read selectively, never flood the context window. This is a performance, cost, and reliability issue. Millions of tokens per run is unsustainable.

3. **Prior art must be consulted** — The original app's patterns and solutions are documented. Check before treating any problem as new.

4. **Research before implementation** — Umbraco's ecosystem (docs, forums, GitHub, Our Umbraco, community blogs) has answers. Don't burn tokens iterating on guesses when the community has already solved it.

5. **Integration mindset** — AgentRun is a plugin into a complex, mature CMS with 20 years of community patterns, undocumented behaviours, and infinite configuration permutations. This is not a greenfield build.

## Team Agreements (carry forward to all future epics)

1. **Context Efficiency Principle** — Agents write intermediate results to disk files. Subsequent steps read selectively. Raw content (HTML, large text, full content trees) must NEVER accumulate in the conversation context. This applies to every workflow and every tool. Performance, cost, and reliability all depend on it.

2. **Research Before Implementation** — When hitting an Umbraco-specific issue or unfamiliar API surface, STOP. Search the Umbraco documentation, forums, GitHub issues, Our Umbraco, and community blogs before attempting a fix. Don't burn tokens iterating on guesses when someone in the community has already solved it.

3. **Integration Mindset** — AgentRun integrates into a complex, mature CMS. Every story accounts for the gap between developer assumptions and real-world editor usage. Test against realistic content with messy, real-world data — not clean starter kits.

4. **Efficiency Over Elegance** — Ship the pragmatic solution that works. No speculative abstractions, no framework-building, no over-engineering. First-mover advantage depends on speed and quality, not architectural perfection.
