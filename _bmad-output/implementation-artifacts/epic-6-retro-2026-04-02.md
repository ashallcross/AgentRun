## Challenges & Growth Areas

### 1. Interactive mode was primary but specs defaulted to autonomous
The PRD states "Human-in-the-loop is the safe, impressive default" but story specs from Epics 2-6 used autonomous language: "Running..." subtitles, spinning icons, Cancel buttons, status badges. This produced the wrong UX across the entire app. Story 6.5 corrected some of it but autonomous-era labels remain throughout.

**Root cause:** Story specs described *how features work* without anchoring to *who's driving*. The question "is the human or the agent in control?" was never asked at the spec level.

**Impact:** Unplanned Story 6.5, plus residual UX debt requiring a cleanup story (7.0).

### 2. Frontend craft quality insufficient
Layout breaks (CSS Grid issue in 6.5), typography/spacing issues, and general polish are below product quality. The team builds functionally correct components without frontend design polish. Every fix is reactive — Adam finds it in the browser, feeds it back, gets patched.

**Root cause:** Backend-heavy team building Lit components without frontend specialist review.

**Action:** Capture as debt for pre-packaging sweep. Adam flagged need for "a proper front-ender to tidy it up."

### 3. Story 6.2 overloaded — three major concerns in one story
Streaming migration (ToolLoop) + ConversationRecorder (new Engine interface + Services implementation) + full chat panel UI (3 new Lit components). Resulted in 7 review patches and 4 deferred items.

**Root cause:** The story that introduces a new UI surface tends to also include the backend work that feeds it.

### 4. Lit reactivity (immutable updates) rediscovered across 3 stories
Direct mutation of ChatMessage objects broke rendering in 6.2 (P3 fix). Same pattern rediscovered in 6.3 (ToolCallData within messages) and 6.4 (user messages). Should have been locked down as a hard rule after first occurrence.

**Root cause:** No project-level documentation of the immutable state requirement for Lit reactivity.

### 5. ToolLoop interactive behaviour retrofitted across 6.4-6.5
ConcurrentQueue couldn't support blocking waits (6.4 addendum → Channel\<string\> refactor). Completion check only ran after ToolLoop exit (6.5 bug fix 3). input.wait SSE event needed for frontend to know when agent is idle (6.5 bug fix 1). All three should have been designed into the ToolLoop from Epic 4.

**Root cause:** Autonomous-first design assumptions. The ToolLoop was designed for "agent runs to completion" not "agent and human take turns."

## Key Insights

1. **"Who's driving?" is a spec-level question, not an implementation detail** — Specs that don't anchor to the primary user mode produce the wrong UX. Interactive-first must be stated explicitly in every story with UI work.
2. **The architecture is sound, the journey was expensive** — The streaming ToolLoop, Channel pipeline, and completion check are the right pieces. But building autonomous-first and retrofitting interactive caused unplanned scope (Story 6.5), addendum scope (6.4 Channel refactor), and 4 E2E bug fixes.
3. **Frontend quality needs a different skillset** — Functional correctness ≠ product quality. Layout, typography, and interaction polish require frontend craft that the current team applies reactively, not proactively.
4. **Immutable state is a framework rule, not a per-story discovery** — Lit reactivity requirements must be documented as hard project rules, not rediscovered in review.
