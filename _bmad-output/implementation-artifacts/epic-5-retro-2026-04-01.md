## What Was Built

- **ToolExecutionException** — structured error type for tool failures, caught by ToolLoop and returned to LLM
- **PathSandbox** — static helper for path validation with traversal prevention and symlink detection, cross-platform
- **ReadFileTool / WriteFileTool / ListFilesTool** — file I/O tools sandboxed to instance folder with atomic writes
- **INetworkAccessPolicy + DefaultNetworkAccessPolicy** — IP blocklist for SSRF protection (RFC1918, loopback, link-local + additional ranges)
- **SsrfProtection** — DNS-resolution-based SSRF validator with testable IDnsResolver abstraction
- **FetchUrlTool** — HTTP GET with SSRF protection, content-type-based size limits, timeout handling
- **ToolDeclaration** — declaration-only AIFunctionDeclaration subclass replacing executable AIFunction (unplanned, emerged from E2E — fixed FunctionInvokingChatClient double-execution)
- **IWorkflowTool.ParameterSchema** — JSON schema property enabling LLM to see tool parameters (unplanned, emerged from E2E)
- **WorkflowRegistry tool validation** — validates step tool references against registered tools at load time, rejects workflows with unknown tools

## Challenges & Growth Areas

### 1. Security code defaulted to allow in unspecified cases
- 13 security-relevant review findings across Stories 5.2 and 5.3
- DefaultNetworkAccessPolicy missing 4 IP ranges (0.0.0.0/8, fc00::/7, 100.64.0.0/10, 240.0.0.0/4)
- Unknown address families defaulted to allow (should deny)
- Empty DNS result silently passed validation
- PathSandbox initially coupled to ToolExecutionException (violated reusability AC)

**Root cause:** Story specs didn't state default posture for unrecognised inputs. Dev agent chose permissive default when unspecified.

**Action taken:** New team agreement — security code must state "deny by default" explicitly in specs.

### 2. Unit tests gave false confidence on integration behaviour
- 252 green backend tests, but runtime was broken until E2E caught it
- FunctionInvokingChatClient double-execution invisible to unit tests
- Empty parameter schemas invisible to unit tests (mocks don't call real LLM)

**Root cause:** Unit tests validate component behaviour in isolation. Integration seam bugs require end-to-end testing with real providers.

**Action reinforced:** Live provider testing remains non-negotiable for integration stories.

## Key Insights

1. **The spec is still the lever** — Failure & Edge Cases reduced critical catches by 87% (8→1). But security code needs an additional rule: state the default posture explicitly.
2. **Defence in depth architecture is sound** — tool whitelisting (workflow.yaml) → sandbox boundaries (PathSandbox/SsrfProtection) → Umbraco permissions (backoffice auth). Balance security with usefulness by making powerful actions opt-in via explicit workflow declaration.
3. **E2E with real provider is non-negotiable** — caught critical bugs in 2 consecutive epics that unit tests couldn't find.
4. **Unplanned work from E2E is valid** — ToolDeclaration and ParameterSchema weren't in any story spec but were the most impactful changes in the epic. Live testing reveals what planning can't predict.
