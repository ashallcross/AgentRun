## Challenges & Growth Areas

### 1. Dev agent overcomplicates bug fixes
Observed across multiple stories — the dev agent tries many approaches instead of the simplest, most robust solution. Adam had to intervene to simplify.

**Root cause:** Agent lacks the browser feedback loop that Adam has. It guesses instead of verifying.

**Action:** Always try simplest solution first. Use Adam's browser testing as a shortcut — ask for verification rather than iterating blindly.

### 2. Planning assumptions proven wrong (4 instances)
- Template name was `umbraco-extension`, not `umbracopackage-rcl` (Story 1.1)
- Bare `lit` imports don't work in Bellissima — must use `@umbraco-cms/backoffice/external/lit` (Story 1.3)
- `sectionSidebarApp` kind `"menuItem"` doesn't exist — must be `"menu"` (Story 1.3)
- Cookie auth doesn't work for Umbraco backoffice API calls — needs bearer token (Story 2.3)

**Impact:** Corrected during implementation with no rework. Process handles it, but integration validation during planning could reduce surprises.

### 3. Deferred work accumulating (25 items)
Key clusters:
- **TOCTOU race conditions** (3.1, 3.2, 3.4) — per-instance locking needed when step executor introduces concurrent callers
- **Frontend testing constraints** — can't render actual Lit components in web-test-runner (Umbraco backoffice exports not importable)
- **Mutable WorkflowDefinition in singleton registry** — no one mutates it today, but it's a latent risk
- **State machine enforcement** — any status transition accepted currently

## Key Insights

3. **Review catches what planning misses** — 3-layer adversarial review is the right investment for a security-conscious package
4. **Deferred work is manageable when tracked explicitly** — the deferred-work.md file prevents items from being forgotten
